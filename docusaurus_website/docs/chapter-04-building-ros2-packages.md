# Chapter 4: Building ROS 2 Packages

## Learning Objectives
- Create a well-structured ROS 2 workspace and understand its directory layout
- Generate new ROS 2 packages using `ros2 pkg create` for both Python and C++ projects
- Configure `setup.py` (Python packages) and understand when `CMakeLists.txt` is used instead
- Write and use ROS 2 launch files to start multiple nodes with configurable parameters
- Build and test ROS 2 packages using the colcon build system

## Introduction

Writing individual ROS 2 nodes is a valuable skill, but deploying a real robot system requires organizing those nodes into packages, configuring them for different environments, and launching them reliably as a coordinated system. ROS 2's build and packaging system provides the infrastructure for all of this — and understanding it well separates engineers who can run tutorials from engineers who can build production systems.

A ROS 2 package is the fundamental unit of software organization in the ROS ecosystem. A package can contain nodes, message definitions, service definitions, configuration files, launch files, and tests. Packages can depend on other packages and are built with colcon, a meta-build tool that orchestrates CMake (for C++ packages) or setuptools (for Python packages). Understanding packages means you can share code between projects, manage dependencies explicitly, and deploy reproducible robot software stacks.

This chapter takes you from an empty workspace to a running multi-node humanoid robot system, covering every step of the process with working commands and code. By the end, you will have a package structure that you can use as a template for your own projects throughout this course.

## Core Concepts

### Workspace Structure

A ROS 2 workspace is a directory containing one or more packages organized under a `src/` subdirectory. Colcon builds all packages in `src/` and places build artifacts, install files, and log files in sibling directories.

```
ros2_ws/                    # The workspace root (name it anything)
├── src/                    # Source packages live here
│   ├── humanoid_control/   # A Python package
│   ├── humanoid_msgs/      # A message-only package
│   └── humanoid_bringup/   # A launch-only package
├── build/                  # Auto-generated by colcon (do not edit)
├── install/                # Installed files (sourced to find packages)
└── log/                    # Build and test logs
```

You should never manually edit the `build/`, `install/`, or `log/` directories. They are fully managed by colcon and can be deleted and regenerated at any time with `colcon build`.

After building, you activate the workspace by sourcing the setup file:

```bash
source install/setup.bash
```

This adds your package's executables, libraries, and Python modules to the appropriate environment variables so ROS 2 can find them.

### Package Types: Python vs. C++

ROS 2 supports two primary build systems for packages:

**Python packages** (using `ament_python` build type):
- Build configuration in `setup.py` and `package.xml`
- Best for: nodes written in Python, launch files, scripts, configuration
- No compilation step — Python files are installed directly
- Faster iteration: change a Python file, `colcon build --symlink-install`, and the change is immediately active without rebuilding

**C++ packages** (using `ament_cmake` build type):
- Build configuration in `CMakeLists.txt` and `package.xml`
- Best for: performance-critical nodes, hardware drivers, custom message definitions
- Requires compilation with CMake
- Generally runs 2–10x faster than equivalent Python at runtime

Most real robot systems use both: Python for high-level logic (planning, state machines, user interfaces) and C++ for low-level control loops and hardware drivers. This chapter focuses on Python packages, which are the right starting point for developers new to robotics.

### The `package.xml` File

Every ROS 2 package has a `package.xml` file that declares the package's metadata and dependencies. ROS 2 uses this file to resolve build-time and run-time dependencies.

```xml
<?xml version="1.0"?>
<?xml-model href="http://download.ros.org/schema/package_format3.xsd"
            schematypens="http://www.w3.org/2001/XMLSchema"?>
<package format="3">
  <name>humanoid_control</name>
  <version>0.1.0</version>
  <description>
    High-level control nodes for a humanoid robot demonstration system.
  </description>
  <maintainer email="dev@example.com">Your Name</maintainer>
  <license>Apache-2.0</license>

  <!-- Build tool dependency: always required for Python packages -->
  <buildtool_depend>ament_python</buildtool_depend>

  <!-- Runtime dependencies: packages your nodes import -->
  <depend>rclpy</depend>
  <depend>sensor_msgs</depend>
  <depend>geometry_msgs</depend>
  <depend>std_srvs</depend>
  <depend>std_msgs</depend>

  <!-- Test dependencies: only needed when running tests -->
  <test_depend>ament_copyright</test_depend>
  <test_depend>ament_flake8</test_depend>
  <test_depend>ament_pep257</test_depend>
  <test_depend>pytest</test_depend>

  <export>
    <build_type>ament_python</build_type>
  </export>
</package>
```

### The `setup.py` File (Python Packages)

For Python packages, `setup.py` tells colcon how to install the package's Python modules and register executable entry points.

```python
# setup.py
from setuptools import setup, find_packages
import os
from glob import glob

package_name = 'humanoid_control'

setup(
    name=package_name,
    version='0.1.0',
    # find_packages() automatically discovers all Python packages (directories
    # with __init__.py) inside this source directory
    packages=find_packages(exclude=['test']),
    data_files=[
        # Install the package.xml so ROS 2 can find the package
        ('share/ament_index/resource_index/packages',
         ['resource/' + package_name]),
        ('share/' + package_name, ['package.xml']),
        # Install all launch files into the share directory
        (os.path.join('share', package_name, 'launch'),
         glob(os.path.join('launch', '*.launch.py'))),
        # Install all config (YAML parameter) files
        (os.path.join('share', package_name, 'config'),
         glob(os.path.join('config', '*.yaml'))),
    ],
    install_requires=['setuptools'],
    zip_safe=True,
    maintainer='Your Name',
    maintainer_email='dev@example.com',
    description='High-level control nodes for humanoid robot',
    license='Apache-2.0',
    tests_require=['pytest'],
    entry_points={
        # console_scripts entries create executable commands that ROS 2 can
        # launch with 'ros2 run <package_name> <executable_name>'
        # Format: 'executable_name = module.path:function_name'
        'console_scripts': [
            'joint_state_publisher = humanoid_control.joint_state_publisher:main',
            'joint_monitor = humanoid_control.joint_monitor:main',
            'gait_controller = humanoid_control.gait_controller:main',
        ],
    },
)
```

## Hands-On: Code Example

### Step 1: Create the Workspace and Package

```bash
# Create workspace directory
mkdir -p ~/ros2_ws/src
cd ~/ros2_ws/src

# Create a Python package with rclpy dependency
# --build-type ament_python: Python package (not C++)
# --dependencies: auto-add these to package.xml
ros2 pkg create humanoid_control \
    --build-type ament_python \
    --dependencies rclpy sensor_msgs geometry_msgs std_srvs std_msgs

# The generated directory structure looks like:
# humanoid_control/
# ├── humanoid_control/      # Python module (same name as package)
# │   └── __init__.py        # Makes it a Python package
# ├── resource/
# │   └── humanoid_control   # Ament index marker file
# ├── test/
# │   ├── test_copyright.py
# │   ├── test_flake8.py
# │   └── test_pep257.py
# ├── package.xml
# └── setup.py

# Create additional directories for launch files and config
mkdir -p ~/ros2_ws/src/humanoid_control/launch
mkdir -p ~/ros2_ws/src/humanoid_control/config

# Verify the package was created correctly
ls ~/ros2_ws/src/humanoid_control/
```

### Step 2: Add Node Source Files

```bash
# Place your node files inside the Python module directory
# (the one with __init__.py, same name as the package)
cp joint_state_publisher.py ~/ros2_ws/src/humanoid_control/humanoid_control/
cp joint_monitor.py ~/ros2_ws/src/humanoid_control/humanoid_control/
```

### Step 3: Create a Parameter Configuration File

```yaml
# ~/ros2_ws/src/humanoid_control/config/robot_params.yaml
# ROS 2 parameter files use YAML with the format:
# /<node_name>:
#   ros__parameters:
#     <param_name>: <value>

/joint_state_publisher:
  ros__parameters:
    publish_rate: 100.0          # Hz — joint state publish frequency
    robot_description: "humanoid_demo"

/joint_monitor:
  ros__parameters:
    effort_warn_threshold: 70.0  # Nm
    effort_crit_threshold: 90.0  # Nm
    log_interval_messages: 100   # Log summary every N messages
```

### Step 4: Create a Launch File

Launch files are Python scripts that describe how to start a system of nodes. They replace ROS 1's XML launch files with a more expressive Python API.

```python
# ~/ros2_ws/src/humanoid_control/launch/humanoid_demo.launch.py
"""
Launch file for the humanoid robot demonstration system.

Launch files are Python modules that return a LaunchDescription.
They allow:
  - Starting multiple nodes as a coordinated system
  - Passing parameters from YAML files
  - Setting remappings (renaming topics/services)
  - Conditional logic (launch different nodes in simulation vs. hardware)
  - Composable nodes (multiple nodes in a single process)

Run with: ros2 launch humanoid_control humanoid_demo.launch.py
"""

import os
from ament_python_utils.get_package_share_directory import get_package_share_directory
from launch import LaunchDescription
from launch.actions import DeclareLaunchArgument, LogInfo, TimerAction
from launch.substitutions import LaunchConfiguration, TextSubstitution
from launch_ros.actions import Node


def generate_launch_description():
    """
    This function MUST be named generate_launch_description().
    ROS 2 calls this function to get the LaunchDescription.
    """

    # ── Locate the package's installed share directory ────────────────────────
    # After 'colcon build', package files are installed to:
    # ~/ros2_ws/install/humanoid_control/share/humanoid_control/
    pkg_share = get_package_share_directory('humanoid_control')
    config_file = os.path.join(pkg_share, 'config', 'robot_params.yaml')

    # ── Declare launch arguments ──────────────────────────────────────────────
    # Launch arguments can be passed on the command line:
    # ros2 launch humanoid_control humanoid_demo.launch.py use_sim:=true
    use_sim_arg = DeclareLaunchArgument(
        'use_sim',
        default_value='false',
        description='Set to true when running in simulation (uses sim time)'
    )

    publish_rate_arg = DeclareLaunchArgument(
        'publish_rate',
        default_value='100.0',
        description='Joint state publish rate in Hz'
    )

    # ── Node configurations ───────────────────────────────────────────────────

    joint_publisher_node = Node(
        package='humanoid_control',        # Which ROS 2 package
        executable='joint_state_publisher', # Which entry_point in setup.py
        name='joint_state_publisher',       # Override node name (optional)
        namespace='/robot',                 # All topics will be /robot/joint_states etc.
        
        # Load parameters from YAML file AND override specific ones from CLI
        parameters=[
            config_file,                    # Load all params from YAML
            {
                # Override publish_rate with the launch argument value
                'publish_rate': LaunchConfiguration('publish_rate'),
                # use_sim_time must be true when using a simulation clock
                'use_sim_time': LaunchConfiguration('use_sim'),
            }
        ],
        
        # Remappings: rename topics at launch time without changing code
        # Format: ('original_topic', 'new_topic')
        remappings=[
            ('/joint_states', '/robot/joint_states'),
        ],
        
        # Output: 'screen' prints node output to terminal
        # Use 'log' to send to log files only (production systems)
        output='screen',
        
        # emulate_tty: ensures colored output and proper line buffering
        emulate_tty=True,
    )

    joint_monitor_node = Node(
        package='humanoid_control',
        executable='joint_monitor',
        name='joint_monitor',
        namespace='/robot',
        parameters=[
            config_file,
            {'use_sim_time': LaunchConfiguration('use_sim')},
        ],
        remappings=[
            ('/joint_states', '/robot/joint_states'),
        ],
        output='screen',
        emulate_tty=True,
    )

    # ── Delayed start ─────────────────────────────────────────────────────────
    # Start the monitor 2 seconds after the publisher to ensure
    # the publisher is ready before the monitor begins subscribing.
    # (In production, use lifecycle nodes for this — see advanced chapters.)
    delayed_monitor = TimerAction(
        period=2.0,  # seconds
        actions=[joint_monitor_node]
    )

    # ── Assemble and return the LaunchDescription ─────────────────────────────
    return LaunchDescription([
        use_sim_arg,
        publish_rate_arg,
        LogInfo(msg='Starting Humanoid Robot Demo System...'),
        joint_publisher_node,
        delayed_monitor,
        LogInfo(msg='All nodes launched. Monitor will start in 2 seconds.'),
    ])
```

### Step 5: Build and Run

```bash
# Navigate to workspace root (NOT src/)
cd ~/ros2_ws

# Source ROS 2 base installation first
source /opt/ros/humble/setup.bash   # Adjust for your ROS 2 distro

# Build all packages in src/
# --symlink-install: Python files are symlinked rather than copied —
#   changes to .py files take effect immediately without rebuilding
colcon build --symlink-install

# If you only want to build one package (faster during development):
colcon build --symlink-install --packages-select humanoid_control

# Source the workspace install to make your packages available
source install/setup.bash

# Verify the package is found by ROS 2
ros2 pkg list | grep humanoid

# Verify your executables are registered
ros2 pkg executables humanoid_control

# Launch the full system
ros2 launch humanoid_control humanoid_demo.launch.py

# Or launch with custom arguments
ros2 launch humanoid_control humanoid_demo.launch.py \
    publish_rate:=50.0 \
    use_sim:=false

# In a separate terminal (after sourcing workspace):
source ~/ros2_ws/install/setup.bash

# Monitor the system
ros2 node list
ros2 topic list
ros2 topic hz /robot/joint_states    # Verify actual publish rate
ros2 topic echo /robot/joint_states  # Print message contents

# Inspect parameter values at runtime
ros2 param list /robot/joint_state_publisher
ros2 param get /robot/joint_state_publisher publish_rate

# Change a parameter at runtime (takes effect immediately)
ros2 param set /robot/joint_state_publisher publish_rate 200.0
```

### Step 6: Colcon Build Tips

```bash
# Build and immediately run tests
colcon build --symlink-install && colcon test

# View test results
colcon test-result --verbose

# Clean the build (when you need a fresh start)
# Be careful: this deletes all build artifacts
rm -rf build/ install/ log/
colcon build --symlink-install

# Build in release mode (C++ packages — enables optimizations)
colcon build --cmake-args -DCMAKE_BUILD_TYPE=Release

# List all packages in the workspace
colcon list

# Show the dependency graph
colcon graph

# Build only packages that changed since last build
colcon build --symlink-install --packages-up-to humanoid_control
```

## Common Mistakes

1. **Building from the `src/` directory instead of the workspace root**: Colcon must be run from the workspace root (the directory containing `src/`, `build/`, `install/`, `log/`). Running `colcon build` from inside `src/` creates nested `build/install/log` directories inside `src/`, polluting your source tree. Always `cd ~/ros2_ws` before running colcon.

2. **Forgetting to source the workspace after building**: After every `colcon build`, you must run `source install/setup.bash` in each open terminal. New terminal sessions do not automatically have your workspace sourced. A common productivity improvement is to add `source ~/ros2_ws/install/setup.bash` to your `~/.bashrc`, but be aware this can cause confusion when switching between multiple workspaces.

3. **Not listing all dependencies in `package.xml`**: If your Python node imports `rclpy` or any ROS 2 message package, that package must appear in `package.xml` as a `<depend>`. Packages not listed in `package.xml` will not be installed when someone builds your package from scratch (e.g., on CI or a new machine), causing import errors that are hard to debug remotely.

4. **Putting node source files in the wrong directory**: For a Python package named `humanoid_control`, node files must be inside `humanoid_control/humanoid_control/` (the Python module directory with `__init__.py`), not in the package root. Files placed in the wrong location will not be discovered by Python and will cause `ModuleNotFoundError`.

5. **Launch files not updating after edits**: Even with `--symlink-install`, launch files in the `launch/` directory are only symlinked if you included them in `setup.py`'s `data_files` section. If you add a new launch file but forget to update `setup.py`, it will not be installed and `ros2 launch` will not find it. Always check your `data_files` glob patterns after adding new file types.

## Summary

- A ROS 2 workspace has a strict `src/build/install/log` structure; `colcon build` is always run from the workspace root, never from inside `src/`.
- Python packages use `ament_python` build type with `setup.py` for entry points and `package.xml` for dependency declaration; C++ packages use `ament_cmake` with `CMakeLists.txt`.
- `setup.py`'s `data_files` section must explicitly include launch files, config files, and any non-Python assets — they are not installed automatically.
- Launch files are Python scripts returning a `LaunchDescription`; they compose nodes, pass parameters from YAML files, accept command-line arguments, and support remappings.
- `colcon build --symlink-install` is the standard development build command; symlink install means Python file edits take effect without rebuilding, dramatically speeding up the development loop.
